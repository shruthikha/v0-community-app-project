# Resident Requests: Admin Guide

As a community administrator, the Resident Requests dashboard allows you to manage community infrastructure, address resident concerns, and maintain transparent communication.

## Requests Dashboard

The admin overview is divided into two main tabs to help you prioritize your workload.

![Admin Overview](pathname:///screenshots/requests_admin_overview_1.png)

*   **Active**: Requests that are `Pending` or `In Progress`. These require your immediate attention.
*   **Resolved**: A history of `Resolved` or `Rejected` requests for record-keeping and audit trails.

## Managing a Request

Click on any request in the table to view the full details, history, and conversation thread.

### Changing Status
Use the sidebar to update the request lifecycle:
*   **Pending -> In Progress**: Let the resident know you have seen the request and are working on it.
*   **In Progress -> Resolved**: Confirm the work is complete.
*   **Rejected**: If a request cannot be fulfilled, provide a clear reason. The resident will be notified.

![Admin Details](pathname:///screenshots/requests_admin_details_sidebar_1.png)

### Internal Notes vs Public Replies
*   **Internal Notes**: (In Development) Secure notes for coordination between admins.
*   **Public Replies**: Any comment you add to the thread is visible to the resident (and the community if the request is public). Use this to request more info or provide status updates.

## Reopening Requests

If a resolved issue returns or requires further work, you can **Reopen** the request. This moves it back to the "Active" tab and notifies the resident that the work has resumed.

## Best Practices
*   **Prompt Initial Response**: Move requests to "In Progress" quickly to reassure residents their voice is being heard.
*   **Use Photos**: Reference resident photos to verify issues before sending maintenance teams.
*   **Contextual Rejection**: Always provide a helpful reason if rejecting a request to maintain community trust.
